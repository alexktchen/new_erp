# Pilot（試營運）選店策略

> **Pilot = 試營運**：先選 10 家加盟店**試跑 2-4 週 ~ 2 個月**，驗證系統 + 流程 + UX，再擴大到其他 90 家。
> 驗證通過前、**總部不強迫**其他加盟店切換。

---

## 1. Pilot 目標優先序（使用者裁定 2026-04-23）

| 優先 | 目標 | 成功條件 |
|---|---|---|
| 🥇 A | **確認系統穩定**（不當機、不掉單、不算錯錢） | Pilot 期內 0 次嚴重 bug、0 筆資料錯誤 |
| 🥈 B | **抓出沒想到的流程漏洞** | 收集 10+ 個新情境需求、能寫成 v0.3 PRD |
| 🥉 C | **店長用得順**（UI 友善） | 店長不需要工程師每天接電話（>80% 問題可自解） |
| ❌ D | ~~示範給其他加盟主看~~ | 不是 pilot 目標（可留給擴張期）|

## 3. 選店框架（10 家怎麼挑）

### 3.1 篩選池：100 - 3（排除）= 97 家候選

### 3.2 必要條件（全部符合才能納入 pilot）

- ✅ 所在地 = **桃園 / 新北 / 台北**
- ✅ 營運穩定（非排除名單）
- ✅ 加盟主配合度 = **積極 或 中立**（不要被動型）
- ✅ 店長技術程度 = 堪用（目前全都符合）
- ✅ 願意簽 pilot MOU（暫訂 2-4 週、最長 2 個月）

### 3.3 優先順序

1. 🥇 **積極派 10 家**全部考慮 — pilot 最需要**願意忍耐 bug + 給反饋**的老闆
2. 🥈 若積極派不到 10 家符合地理條件 → 從 **中立派**補、優先挑**桃園**（總部支援最輕鬆）
3. 🥉 **絕對要有 2-3 家桃園店**當「保底」— 遠端搞不定時可殺去現場
4. 🥈 中立派挑選時優先挑「訂單量中等」的店 — 避免流量最大（風險高）或最小（代表性低）

### 3.4 建議組成（10 家）

| 區域 | 家數 | 備註 |
|---|---|---|
| **桃園** | **4 家**（2 積極 + 2 中立） | 含保底站、總部近、可現場支援 |
| **新北** | **3 家**（2 積極 + 1 中立） | 約 30 分鐘車程可到 |
| **台北** | **3 家**（積極為主） | 最遠、純遠端支援 |

**流程複雜度分佈**（B 目標）：
- 2 家選「退貨多 / 員工團購多」的店 — 專門挖流程漏洞
- 2 家選「跨店調貨多」的店 — 測試加盟店之間互助
- 其他 6 家選「中規中矩」— 建立 baseline

## 4. Pilot 時程 — 兩種方案

### 方案 AA — 激進版（你選的、5 月底上線）

| 週次 | 日期 | 事件 |
|---|---|---|
| W1 | 4/23-4/30 | 選 10 家 + 跟加盟主談 MOU + 簽約 |
| W2-3 | 5/1-5/15 | **工程衝刺**：RPC / 管理後台 / ezPay 子商店 API 基本可用 |
| W4 | 5/16-5/22 | 店長訓練（線上 + 圖文）+ 資料遷移 |
| W5-8 | 5/23-6/20 | **Pilot 跑 4 週** |
| W9 | 6/21-6/27 | 評估、決定擴大或延期 |

**硬核前提**（缺任一項 → 方案 AA 不可行）：
- 🚨 需要**工程師 full-time 寫 3 週**（或你自己寫）
- 🚨 PR #101 admin console **立即 merge + deploy**
- 🚨 ezPay 子商店 API **5 月初確認可用**（pilot 店若達 20 萬門檻要能切 `enabled`）
- 🚨 店長訓練教材 **兩週內產出**（圖文、20 分鐘短影片）

### 方案 BB — 安全版（暑假 pilot）

| 月份 | 事件 |
|---|---|
| 5 月 | 工程 sprint、完成 RPC + UI |
| 6 月 | 內部測試 + 選店談判 + 訓練教材 |
| 7/1 | Pilot 10 家上線 |
| 7/1-8/31 | Pilot 跑 **2 個月（一期發票週期）** |
| 9 月初 | 評估、決定擴大到 30/50 家 |
| 10-12 月 | 滾動擴大到 100 家 |

**優點**：
- 有充分測試時間
- 涵蓋一個完整發票期（Q17 per-store invoice mode 能驗證）
- 擴張時已累積 2 個月實戰反饋

## 5. 成功指標（Go / No-Go 決策標準）

Pilot 結束時、用這些指標決定是否擴大：

### ✅ 全綠（Go 擴大到 30-50 家）
- 0 筆資料錯誤（金額、庫存、訂單對得起來）
- 0 次嚴重系統停擺（> 4 小時無法使用）
- 店長日常操作 **問題率 < 1/店/週**
- 加盟主**願意繼續用**（8 家以上繼續）
- 收集到 ≥ 10 個具體流程改進建議

### ⚠️ 黃燈（修好再擴）
- 1-2 筆資料錯誤（有原因、已修復）
- 1-2 次短暫停擺（< 2 小時）
- 店長問題率 1-3/店/週
- 6-8 家願意繼續

### 🚨 紅燈（Pull back、延期擴大）
- ≥ 3 筆無法還原的資料錯誤
- ≥ 1 次長時間停擺（> 8 小時）
- 店長問題率 > 5/店/週、接電話接不完
- < 5 家願意繼續
- **回退方案**：Pilot 10 家退回舊 lt-erp / 紙筆、systme v0.3 重構

## 6. Pilot 期間的支援機制

- 📞 **LINE 群組**：每家 1 店 + 總部、即時問答（24h 內回）
- 📋 **問題 log**：每家店收到的問題都記在 GitHub Issues、**不在 LINE 散失**
- 🚗 **桃園現場支援**：總部可 1 小時內到場（最多 3-4 次/月）
- 📊 **週會**：每週五總部 + 10 家店長 30 分鐘同步（視訊）
- 📖 **FAQ 不斷更新**：第 5 次有人問同樣問題 = 進 FAQ

## 8. 反模式（絕對不要做）

- ❌ 一次讓 100 家都上線（pilot 的意義就是不這樣）
- ❌ 選不配合的店硬推（被動派 / 反對派）
- ❌ Pilot 期間**沒有退出機制**（加盟主想退必須能退回舊流程）
- ❌ 忽略 pilot 反饋、照原 PRD 硬推擴張（pilot 是為了**改 PRD** 的）
- ❌ 把 pilot 當「示範」給還沒用的加盟主看（那是擴張期的事、不是 pilot 期目標）
